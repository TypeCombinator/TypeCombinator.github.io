## std::expected

Using `[[no_unique_address]]` to reuse the tail padding introduced by memory alignment in non-C structs. Consider the following code:

```c++
class Foo {
    int i;
    char c;
    bool b;
};
enum class ErrCode : int { Err1, Err2, Err3 };
```

Due to memory alignment, there are 2 bytes of padding at the end (**keep this in mind**), so `sizeof(Foo)` is 8. What is `sizeof(std::expected<Foo, ErrCode>)` then? With libstdc++, the result is 12, but with libc++, it's 8. Why? The libc++ simplified `std::expected` code looks like this:

```c++
template <class Val, class Err>
class expected {
    union U {
        [[no_unique_address]] Val val_;
        [[no_unique_address]] Err err_;
    };
    [[no_unique_address]] U u_;
    bool has_val_;
};
```
![](/blog/cppcon2025_huixie90/expected0.png)

Because *"The attribute-token no_unique_address specifies that a non-static data member is a potentially-overlapping subobject"*, `class Foo` is not a C struct, so its tail padding is reused. The benefit is a smaller memory footprint and better cache locality. `std::expected` is commonly used for error handling and return values. For the following code:

```c++
std::expected<Foo, ErrCode> compute() { return Foo{}; }
```

The assembly generated by gcc with libstdc++ is:

```assembly
compute():
        mov     DWORD PTR [rsp-24], 0
        xor     eax, eax
        mov     BYTE PTR [rsp-24], 1
        mov     ecx, DWORD PTR [rsp-24]
        mov     rdx, rcx
        ret
```

While clang with libc++ produces:

```assembly
compute():
        movabs  rax, 281474976710656
        ret
```

But is it all upside? Imagine your neighbor has a big yard, part of it used for a swimming pool, part for a garden... with some unused space left over. As their neighbor, if you see vacant land and presumptuously place your belongings there, what happens? Naturally, it won't end well between neighbors! Suppose the `std::expected` copy constructor is implemented as follows:

```c++
template <class Val, class Err>
class expected {
  public:
    expected(const expected& other)
        : has_val_(other.has_val_) {
        if (has_val_) {
          std::construct_at(std::addressof(u_.val_), other.u_.val_);
        } else {
          std::construct_at(std::addressof(u_.err_), other.u_.err_);
        }
    }
};
```

At first glance, the above code seems intuitive, but what happens if you try running the following?

```c++
int main() {
    std::expected<Foo, int> e1(Foo{});
    std::expected e2(e1);
    assert(e2.has_value());
}
```

The assertion fails! Because *"To zero-initialize an object or reference of type T means: ... if T is a (possibly cv-qualified) non-union class type, its padding bits ([basic.types.general]) are initialized to zero bits and ..."* . The neighbor says, I'm cleaning my own yard; if I throw out the junk in the empty space, you can't blame me! For `class Foo`, `construct_at` zeros out the padding bits, and `has_val_` gets zeroed along with them. The fix is to move the assignment of `has_val_` after the call to `std::construct_at`, like this:

```c++
template <class Val, class Err>
class expected {
  public:
    expected(const expected& other) {
        if (other.has_val_) {
          std::construct_at(std::addressof(u_.val_), other.u_.val_);
        } else {
          std::construct_at(std::addressof(u_.err_), other.u_.err_);
        }
        has_val_ = other.has_val_;
    }
};
```

So, is the problem fully resolved? Consider nested `std::expected`, like this:

``` c++
std::expected<std::expected<Foo, ErrCode>, ErrCode> e = ...;
e.value().emplace(); // the inner expected construct_at will overwrite outer expected bool
```
At this point, the memory layout of `std::expected` is:

![](/blog/cppcon2025_huixie90/expected1_nest0.png)

Calling `emplace` on the inner `expected` object invokes `construct_at`, which zeros out the padding space, and the outer `expected`'s `has_val_` gets zeroed along with it. This is similar to the copy constructor issue above. A more intuitive illustration:

```c++
struct Bar {
    [[no_unique_address]] std::expected<Foo, ErrCode> e;
    char c;
};

Bar bar = ...;
bar.c = 'c';
bar.e.emplace(); // construct_at will overwrite c
```
The memory layout of `struct Bar` is:

![](/blog/cppcon2025_huixie90/expected1_nest1.png)

The member `char c` reuses the tail padding of the preceding member, and `construct_at` zeros out the memory of member `c`. The character write in the code above becomes a no-op! To solve this, we modify the code once more:

```c++
template <class Val, class Err>
class expected {
    struct repr {
        union U {
            [[no_unique_address]] Val val_;
            [[no_unique_address]] Err err_;
        };
        [[no_unique_address]] U u_;
        bool has_val_;
    };
 
    repr repr_; // no [[no_unique_address]] on this member
};
```

The key is that `repr repr_;` does not have `[[no_unique_address]]`, which breaks the `[[no_unique_address]]` recursion. `char c` will no longer reuse the tail padding. At this point, the memory layout of `struct Bar` is:

![](/blog/cppcon2025_huixie90/expected2.png)

Now, after all these changes, is this the optimal code? What if neither `Val` nor `Err` has any padding? For example, `std::expected<unsigned int, int>`: the member `bool has_val_` would have no tail padding to reuse, and the memory layout would be:

![](/blog/cppcon2025_huixie90/expected3.png)

The final modification adds a branch to handle this case:

```c++
template <class Val, class Err>
struct repr {
    union U {
        [[no_unique_address]] Val val_;
        [[no_unique_address]] Err err_;
    };
    [[no_unique_address]] U u_;
    bool has_val_;
};

template <class Val, class Err>
struct expected_base {
    repr<Val, Err> repr_; // no [[no_unique_address]]
};
template <class Val, class Err> requires bool_is_not_in_padding
struct expected_base {
    [[no_unique_address]] repr<Val, Err> repr_;
};

template <class Val, class Err>
class expected : expected_base<Val, Err> {};
```

Only when the member `bool has_val_` does not reuse padding do we add `[[no_unique_address]]`, a conditional attribute. At this point, the tail padding of `repr<Val, Err> repr_` is fully under control, and the current `expected` does not read or write this tail padding, so it can be safely reused externally.

The following is **my humble opinion**. In the past, achieving potentially-overlapping subobjects was only possible through inheritance. Now, using composition with the `[[no_unique_address]]` attribute achieves the same thing. This means that even a class marked `final` cannot prevent overlapping, and you must not have magic code that reads or writes tail padding. For example, using `std::memset` and `std::memcpy` requires not only trivially-copyable but also that the object is not a potentially-overlapping subobject; otherwise it's UB. For instance, writing code like this with libc++'s `std::expected` would be problematic:

```c++
class Foo final {
    int i;
    char c;
    bool b;
   public:
    void clear() noexcept {
        // *this = {}; // Good, but little bit slow.
        std::memset(this, 0, sizeof(*this)); // Bad, but fast.
    }
};

enum class ErrCode : int { Err1, Err2, Err3 };

static_assert(std::is_trivially_copyable_v<Foo>);

int main() {
    std::expected<Foo, ErrCode> e{Foo{}};
    
    static_assert(sizeof(e) == 8);

    e.value().clear();
    assert(e.has_value());
    return 0;
}
// clang++ -std=c++23 -stdlib=libc++
```

The above is only to illustrate the changes brought by C++20's `[[no_unique_address]]`, not to imply that libc++'s `std::expected` implementation has issues. But what if you have tail padding, don't want it reused, and don't want to manually pad for alignment? The standard doesn't provide an attribute like `[[non_overlapping]]` either! A simple approach that comes to mind is borrowing the transparent comparator pattern: for types you don't want overlapped, add a line like `using non_overlapping = ???;`, and the template class taking this type can detect it with `is_non_overlapping` and handle it accordingly. Or flip the logic?

```c++
template <typename T>
concept is_non_overlapping = requires { typename T::non_overlapping; };
```
But this shifts the burden to library authors, which isn't ideal either. What about wrapping it in an **anonymous struct**? That doesn't conform to the C++ standard. Better to just manually align and stay away from magic code!

## stop_token

`stop_source`, `stop_token`, and `stop_callback` are the three components introduced in C++20 for cooperatively stopping threads. They share a common state, roughly implemented as follows:

```c++
class stop_token {
  std::shared_ptr<__stop_state> state_;
};

class stop_source {
  std::shared_ptr<__stop_state> state_;
};
 
template <class Callback>
class stop_callback {
  [[no_unique_address]] Callback callback_;
  std::shared_ptr<__stop_state> state_;
};
```

The `__stop_state` can be implemented like this:

```c++
class __stop_state {
  std::atomic<bool> stop_requested_;
  std::atomic<unsigned> stop_source_count_; // for stop_possible()
  std::list<stop_callback*> stop_callbacks_;
  std::mutex list_mutex_;
};
static_assert(sizeof(__stop_state) == 72);
static_assert(sizeof(stop_token) == 16);
```

The optimized libc++ implementation looks roughly like this:

```c++
class __stop_state {
  // The "callback list locked" bit implements a 1-bit lock to guard
  // operations on the callback list
  //
  //       31 - 2          |  1                   |    0           |
  //  stop_source counter  | callback list locked | stop_requested |
  atomic<uint32_t> state_ = 0;
 
  // Reference count for stop_token + stop_callback + stop_source
  atomic<uint32_t> ref_count_ = 0;
 
  // Lightweight intrusive non-owning list of callbacks
  // Only stores a pointer to the root node
  __intrusive_list_view<stop_callback_base> callback_list_;
};
static_assert(sizeof(__stop_state) == 16);
static_assert(sizeof(stop_token) == 8);
```

First, every bit of the atomic variable is fully utilized, eliminating the need for `std::mutex`. Then, intrusive `shared_ptr` and intrusive linked list are adopted. The memory footprint of `__stop_state` is drastically reduced, and `stop_token` shrinks to the size of a single pointer.

## ranges

### Segmented Iterators

```c++
std::deque<int> d = ...
// 1
for (int& i : d) {
    i = std::clamp(i, 200, 500);
}
// 2
std::ranges::for_each(d, [](int& i) {
    i = std::clamp(i, 200, 500);
});
```

The `std::ranges::for_each` version of the above code is significantly faster. The reason has to do with `std::deque`'s design: the container is organized in blocks, and `std::ranges::for_each` uses a nested loop that fits `std::deque`'s structure perfectly.
![](/blog/cppcon2025_huixie90/deque.png)


### ranges::copy

```c++
std::vector<std::vector<int>> v = ...;
std::vector<int> out;
// 1
out.reserve(total_size);
for (const auto& inner : v) {
  for (int i: inner) {
    out.push_back(i);
  }
}
// 2
out.resize(total_size);
std::ranges::copy(v | std::views::join, out.begin());
```

`std::ranges::copy` is much faster because, for this scenario, it internally calls `std::memmove`. The code:

```c++
template <class Iter, class OutIter>
  requires is_segmented_iterator<Iter>
pair<Iter, OutIter> __copy(Iter first, Iter last, OutIter result) {
  std::__for_each_segment(first, last, [&](auto inner_first, auto inner_last) {
      result = std::__copy(inner_first, inner_last, std::move(result)).second;
  });
  return std::make_pair(last, std::move(result));
}
template <class In, class Out>
  requires can_lower_copy_assignment_to_memmove<In, Out>
pair<In*, Out*> __copy(In* first, In* last, Out* result) {
  std::memmove(result, first, last - first);
  return std::make_pair(last, result + n);
}
```

### flat_map Insertion

```c++
// flat_map<int, double>
class flat_map {
  std::vector<int> keys_; // always sorted
  std::vector<double> values_;
  [[no_unique_address]] std::less<int> compare_;
};
std::flat_map<int, double> m1 = ...;
std::flat_map<int, double> m2 = ...;
// 1
for (const auto& [key, val] : m2) {
  m1.emplace(key, val);
}
// 2
m1.insert_range(m2);
```

Using `insert_range` is faster, and the code is:

```c++
template<container-compatible-range<value_type> R>
constexpr void insert_range(R&& rg) {
  __append(ranges::begin(rg), ranges::end(rg)); // O(M)
 
  auto zv = ranges::views::zip(keys_, values_);
  ranges::sort(zv.begin() + old_size, zv.end()); // O(MLogM)
 
  ranges::inplace_merge(zv.begin(), zv.begin() + old_size, zv.end()); // O(M+N)
 
  auto dup_start = ranges::unique(zv).begin(); // O(M+N)
  __erase(dup_start); // O(M+N)
}

template <class InputIterator, class Sentinel>
void __append(InputIterator first, Sentinel last) {
  for (; first != last; ++first) {
    std::pair<Key, Val> kv = *first;
    keys_.insert(keys_.end(), std::move(kv.first));
    values_.insert(values_.end(), std::move(kv.second));
  }
}
```

The `__append` in the above code inserts elements one by one using a `for` loop, but it can be made even faster. The author introduced the `product_iterator` concept, with code like this:

```c++
template <class Iter>
  requires is_product_iterator_of_size<Iter, 2>
void __append(Iter first, Iter last) {
  using Traits = product_iterator_traits<Iter>;

  keys_.insert(keys_.end(),
      Traits::template get_iterator<0>(first),
      Traits::template get_iterator<0>(last));
 
  values_.insert(values_.end(),
      Traits::template get_iterator<1>(first),
      Traits::template get_iterator<1>(last));
}
```
